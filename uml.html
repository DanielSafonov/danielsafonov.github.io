UML

02 Jan 2020 | uml architecture
Иметь все диаграммы UML - не обязательно, но некоторые из них здорово упростили бы жизнь сначала разработчикам, а потом и тестировщикам.

UML (Unified Modeling Language — унифицированный язык моделирования) — язык графического описания для объектного моделирования в области разработки программного обеспечения, для моделирования бизнес-процессов, системного проектирования, а также отображения организационных структур.

UML является языком широкого профиля, это — открытый стандарт, использующий графические обозначения для создания абстрактной модели системы. UML основан на базе нескольких предыдущих стандартов, таких как метод Гради Буча (Booch), метод Джима Румбаха (OMT) и метод Айвара Джекобсона (OOSE).

Диаграммы UML

  • Структурные (Structure):
    • Классов (Class diagram)
    • Компонентов (Component diagram)
    • Композитной структуры (Composite structure diagram)
      • Кооперации (Collaboration diagram)
    • Развертывания (Deployment diagram)
    • Объектов (Object diagram)
    • Пакетов (Package diagram)
    • Профилей (Profile diagram)
  • Поведенческие (Behavior):
    • Деятельности (Activity diagram)
    • Состояний (State Machine diagram)
    • Вариантов использования (прецедентов) (Use case diagram)
    • Взаимодействия (Interaction):
      • Коммуникаций (Communication diagram)
      • Обзора взаимодействия (Interaction overview diagram)
      • Последовательностей (Sequence diagram)
      • Синхронизации (Timing diagram)

Все представленные выше диаграммы связаны между собой. Комбинируя их, можно добиться необходимого уровня декомпозиции.

Для себя в качестве необходимого минимума я выделил диаграммы

  • прецедентов (use case diagram)
  • последовательностей (sequence diagram) или деятельности (activity diagram)
  • состояний (state Machine diagram)
  • классов (class diagram)
